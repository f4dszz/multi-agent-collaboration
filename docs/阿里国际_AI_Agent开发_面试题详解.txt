阿里国际 AI Agent 开发 — 面试题详解
==========================================

Q1. 在 Plan-And-Execute 架构中，如果子任务之间存在隐含的语义依赖（比如 A 的结论会微妙影响 B 的判断，但没有直接数据传递），你如何设计来规避逻辑断裂？

A1.

这个问题的本质是：DAG 只建模了显式数据流，但子任务之间还有隐式的认知依赖。

设计方案分三层：

第一层：共享上下文黑板（Shared Blackboard）。所有子任务执行完后不仅输出结构化 result，还要往一个全局黑板写入"副产品"——比如 A 任务得出的中间判断、假设、置信度。B 任务启动时，Planner 自动将黑板上与 B 语义相关的条目注入其 context。这类似于多 Agent 协作中的"共识状态"机制，agent 之间不只传递结论，还传递推理过程中的关键事实。

第二层：依赖图的动态补边。在 Planner 分解任务后，加一轮"依赖审查"——用 LLM 对所有子任务两两检查："A 的结论是否可能影响 B 的判断？"如果是，给 DAG 加一条软依赖边。这条边不传数据，而是传 A 的 summary。代价是 O(n²) 的审查，但在子任务数 <20 时完全可接受。

第三层：执行时的回溯机制。当 B 在执行过程中发现自己的判断与黑板上 A 的某个结论矛盾时，触发一个 contradiction_detected 事件，Orchestrator 决定是否回溯重做 A 或者召集仲裁。这在 Java 微服务里类似 Saga 模式的补偿事务。

Q2. 面对 Long-horizon Reasoning 任务，Agent 容易出现"目标漂移"，除了设置最大循环次数，还有什么办法？

A2.

最大循环次数只是兜底，真正的解法：

1. Goal Anchoring（目标锚定）。每轮推理开始前，强制将原始目标以"你当前要解决的核心问题是：XXX"的形式注入 prompt 的最前面。这不是简单复读，而是让模型每一步都对齐原始意图。实践验证过，把 goal 放在 context 末尾 vs 开头，漂移率差 3 倍。

2. Progress Tracker + 偏离检测。维护一个轻量的进度向量——用 embedding 计算当前输出与原始目标的余弦相似度。如果连续两轮 similarity 下降超过阈值，触发"recalibration step"：暂停执行，让一个独立的 Monitor Agent 判断当前方向是否仍在通往目标。

3. 分层规划 Hierarchical Decomposition。不要让 Agent 一口气推理到底。把 Long-horizon 任务拆成 milestone，每个 milestone 结束后做一次 check-in：当前 milestone 的产出是否仍然服务于最终目标？这跟多 Agent 系统里的"阶段审批门"思路一致——不是一口气跑完，而是阶段性收敛。

4. 工作记忆的选择性遗忘。过长的 context 本身就是漂移的温床。定期压缩历史，只保留与目标相关的关键事实，丢弃探索性的中间尝试。

Q3. 你们的 6-Agent 体系中，如果 Reviewer 和 Executor 由于对业务边界理解不一致产生"协同震荡"（反复打回修订），怎么办？

A3.

根因分析：震荡通常不是因为能力不足，而是因为双方对"什么算完成"的标准不对齐。Executor 认为自己改了，Reviewer 认为没改到位，但双方在描述"到位"时用的是不同的隐含标准。

解法一：共识状态文件（Consensus Document）。维护一个双方都可写的共识文件。每轮结束后双方必须就"当前已达成的共识"和"仍有分歧的点"写到这个文件里。如果一个 issue 连续两轮出现在"分歧"栏里，自动升级。这在实际的执行人-监督人协作中验证有效——69+ 轮决策中震荡次数显著下降。

解法二：仲裁者介入 + 衰减策略。在 workflow policy 里设 revision_threshold=2——如果同一个 finding 两轮打回后仍未解决，自动引入第三方 Verifier 做裁决。Verifier 不重新做 Executor 的工作，只判定"Reviewer 的要求是否合理"。

解法三：Reviewer 的 finding 必须结构化。不允许 Reviewer 说"这里不好"，必须给出：具体位置、问题描述、验收标准。Executor 必须逐条回应。这样震荡会快速收敛，因为双方在同一个结构化框架下对话。

Q4. 在高并发场景下，多个研究任务共用同一组 Agent 集群，如何处理 Agent 级别的状态隔离与资源竞态问题？

A4.

状态隔离：每个研究任务实例化独立的 Agent 运行时上下文（类似 Java 里的 ThreadLocal 或 Go 的 goroutine-local storage）。核心设计：

    AgentInstance = Agent模板 + TaskContext + 独立WorkingMemory

Agent 模板（系统 prompt、工具集）是共享的只读资源。TaskContext 和 WorkingMemory 是 per-task 的。在 Python 里用 dataclass 隔离，在 Java 里用 Spring 的 prototype scope bean。绝不允许 Agent 实例之间共享可变状态。

资源竞态：主要发生在两个地方——LLM 调用的并发限流、共享工具（如数据库查询、文件系统）的互斥。

- LLM 调用：用令牌桶或信号量做并发控制。asyncio.Semaphore(max_concurrent_llm_calls)。关键是区分优先级——交互式任务优先于批处理任务。
- 共享工具：对有副作用的工具（写文件、修改数据库）加分布式锁（Redis RedLock 或者简单的 advisory lock）。对只读工具放开并发。
- 资源池化：Agent 不持有长连接，而是从 pool 中按需获取。用完立即归还。Java 里就是连接池模式，Python 里用 contextmanager。

Q5. 针对金融 PDF 中的跨页表格或嵌套层级，如果 MinerU 解析仍有误差，你如何设计一套纠错链路？

A5.

MinerU 等解析工具的误差在金融文档里几乎不可避免，特别是跨页表格和嵌套表头。设计一个多阶段纠错 pipeline：

阶段一：解析 + 置信度标注。MinerU 解析后，对每个表格单元标注置信度。跨页拼接处、合并单元格、数字密集区域天然低置信度。

阶段二：结构校验。用规则引擎做基本 sanity check——列数是否一致、数字列的 sum 是否等于合计行、百分比列是否加总为 100%。金融表格的这些约束是很强的先验知识。任何不满足约束的行/列标记为 suspect。

阶段三：LLM 修复。只对 suspect 区域调用 LLM。把原始 PDF 的截图（vision）+ MinerU 的解析结果一起给模型："以下是 PDF 原图和解析结果，请修正表格中的错误。"多模态对照修复比纯文本修复准确率高很多。

阶段四：人工兜底 + 反馈闭环。对 LLM 修复后仍不满足约束的，标记为"需人工审核"。人工修正的结果反馈回去做 few-shot example，逐渐提升自动修复率。

Q6. 在混合检索中，BM25 与向量搜索的得分不在一个量级，除了使用 RRF，还有什么办法？

A6.

除了 RRF（Reciprocal Rank Fusion），还有几个实用方案：

1. 分数归一化后线性加权。对 BM25 和向量分数分别做 min-max 归一化到 [0,1] 区间，然后 final_score = α * bm25_norm + (1-α) * vector_norm。α 可以通过一小批标注数据 grid search。简单但有效。

2. 学习型融合（Learn-to-Rank）。把 BM25 分数、向量分数、以及其他特征（chunk 长度、文档时效性、query-doc 词汇重叠率等）作为特征向量，用一个轻量的排序模型（LambdaMART 或小型 MLP）来学习最终排序。训练数据可以从用户点击或 LLM 标注的相关性中获取。

3. 分阶段级联。第一阶段用 BM25 做粗召回（top-200），第二阶段用向量模型做精排（top-20）。两者不融合分数，而是 pipeline 式的。好处是可以单独优化每个阶段。

4. 倒数分布归一化。受 DBSF（Distribution-Based Score Fusion）启发，不看绝对分数，而是看每个候选在各自分布中的分位数。BM25 得分在其分布中排 top 5% 的，和向量得分在其分布中排 top 5% 的，视为同等相关。

Q7. 面对 Lost in the Middle 难题，如果必须召回 Top-50 的 Chunk 才能覆盖证据链，你如何设计方案？

A7.

这是个真实的工程痛点。模型对长 context 中间位置的注意力明显下降。

方案一：证据链感知的重排序。召回 Top-50 后，不是按相关性排序塞进 context，而是：先用 LLM（或小模型）识别出 50 个 chunk 之间的证据依赖关系——"Chunk 3 的结论支撑 Chunk 17 的前提"。然后按照依赖链的逻辑顺序排列，而非按相关性分数排。这样模型读到的是一个有逻辑递进关系的文本流，而不是散乱的碎片。

方案二：Map-Reduce。50 个 chunk 分成 5 组，每组 10 个。第一轮（Map）：分别对每组做 extraction——"从这 10 个 chunk 中提取与问题相关的关键事实"。第二轮（Reduce）：把 5 个 extraction 结果合并，做最终推理。每轮的 context 长度可控，完全避开 Lost in the Middle。

方案三：关键 chunk 前置 + 后置。根据 U 型注意力曲线，把最重要的证据放在 context 的开头和结尾，次要证据放中间。简单但实测有用。

方案四：层级摘要 + 按需展开。先对 50 个 chunk 做层级摘要——每 5 个 chunk 一个段落摘要，10 个摘要再生成一个总摘要。模型先看总摘要决定需要深入哪些部分，再按需展开对应的原始 chunk。这是一种"懒加载"策略。

Q8. 引入的 Reasoning Reward Model，在 GRPO 训练中如何定义"推理步骤"的奖励函数，以确保模型不是在玩文字游戏而是真的在逻辑推导？

A8.

这个问题的核心挑战是：如何区分"看起来像推理"和"真的在推理"。

奖励函数设计三维度：

维度一：逻辑一致性（Process Reward）。对每个中间步骤检查：当前步骤的结论是否能从前一步的前提逻辑推出？用一个验证模型（Verifier）对每一步打分。如果模型跳步或出现 non-sequitur，该步奖励为负。这就是 OpenAI PRM800K 的思路，但需要扩展到你自己的 domain。

维度二：事实锚定性。推理步骤中引用的事实是否可追溯到 source？如果模型说"根据表 3，收入增长了 20%"，检查表 3 里是否真的有这个数据。对有幻觉的步骤施加惩罚。这需要 ground truth 或者 RAG 检索结果作为对照。

维度三：不可约性（Non-triviality）。检测模型是否在"注水"——重复前文、同义替换、不增加新信息的步骤给零奖励。具体实现：计算每一步与已有步骤的语义增量（cosine distance），增量低于阈值则判定为 trivial。

GRPO 中的具体实现：对同一问题采样多条推理路径。用上述三维度对每条路径的每个步骤打分，路径的总奖励是步骤奖励的加权和。GRPO 根据路径间的相对优势更新策略。关键 trick 是：奖励函数本身不应该只看最终答案对不对（ORM），而应该重过程（PRM），否则模型会学到"碰运气蒙对答案"而非真正推理。

Q9. Checkpoint 机制在存储全局状态时，如果遇到大模型输出的不可预测性导致状态不一致，怎么办？

A9.

不一致的典型场景：Checkpoint 记录了 state=plan_review，但模型实际输出了一个不符合预期的 response（比如直接给了 implementation 而非 review），导致 state 和实际产出物不匹配。

方案一：Write-Ahead Log + 幂等恢复。借鉴数据库 WAL 思想。每次状态转移前，先写入意图日志："将从 plan_review 转移到 plan_revision"。执行完毕后写入确认。恢复时：有意图无确认则重试；有确认则跳过。所有操作设计为幂等的。

方案二：状态校验而非状态信任。恢复 checkpoint 后，不直接信任 recorded state，而是做一轮 validation：检查 artifacts 目录下实际存在的文件，与 DB 记录的 state 是否逻辑一致。如果 state 说在 implementing 但 artifacts 里没有 approved plan，说明状态有问题——回退到最近一个"自洽"的 checkpoint。

方案三：Event Sourcing 模式。不存储当前状态快照，而是存储所有状态变更事件的有序序列。恢复时从事件序列重放，天然具备一致性。代价是恢复时间略长，但对于 Agent workflow 这个量级完全可接受。

Q10. 在三层记忆架构中，如果中期摘要丢失了短期记忆中的关键事实，你如何设计来保证推理的真实性？

A10.

问题本质：摘要是有损压缩。压缩过程中会丢失对后续推理至关重要的细节。

方案一：关键事实提取 + 单独存储。在从短期到中期压缩时，不是直接 summarize，而是两步走：(1) 提取关键事实（key facts extraction），以结构化键值对形式单独存储；(2) 对剩余内容做 narrative summary。中期记忆 = key facts dict + narrative summary。后续推理时，key facts 总是被优先注入 context。

方案二：摘要质量的自动校验。压缩后，用一个验证步骤：把原始短期记忆中的核心 claim 列出来，逐条检查是否能从中期摘要中推导出来。不能推导出的 claim，补充进摘要。

方案三：多粒度保留。不做二选一（完整保留 vs 摘要），而是三级：原始文本（短期，滑动窗口淘汰）→ 带关键细节的摘要（中期，保留数小时）→ 纯要点（长期）。推理时根据任务需要从不同层级拉取。如果当前任务涉及精确数字或具体事实，从中期层级拉取带细节的版本。

Q11. 针对跨任务共享情报，如果旧知识已过期，怎么办？

A11.

方案一：TTL + 版本号。每条共享知识带时间戳和版本号。消费方使用前检查：是否超过 TTL？是否有更新版本？过期的知识不删除（可能有参考价值），但标记为 stale，注入 context 时加前缀"[可能已过期]"，让 LLM 自行判断可信度。

方案二：来源追溯 + 主动失效。每条知识记录其来源（哪个任务、哪次分析产生的）。当来源数据被更新（比如新的研报发布），主动触发级联失效——所有依赖该来源的知识标记为待刷新。类似于缓存的主动 invalidation。

方案三：知识的置信度衰减。参考记忆的遗忘曲线，知识的置信度随时间指数衰减。融合使用时，旧知识的权重自动降低。如果新产生的知识与旧知识矛盾，新知识自动覆盖。

Q12. 在对 Reporter 进行 SFT 时，如何平衡"格式遵从度"与"生成创造性"？如果模型严重过拟合，你首选调节哪个超参？

A12.

核心矛盾：格式遵从度要求模型"照模板填空"，创造性要求模型"自由表达有洞察的内容"。过度 SFT 格式会让输出变成八股文。

训练策略：

- 数据配比：训练数据中，70% 是高质量格式规范的样本，30% 是格式略有变化但内容有深度洞察的样本。让模型学到"格式是 soft constraint，内容质量是 hard constraint"。
- Loss 权重：对格式 token（标题、分隔符、固定短语）和内容 token 用不同的 loss 权重。格式部分权重适当降低，避免模型把注意力全花在复制模板上。

过拟合时首选调节的超参：Learning Rate 和训练轮数（epoch）。这是最直接的。SFT 过拟合的典型表现就是训练太久或 LR 太大——模型记住了训练样本的具体措辞而非格式规律。先把 epoch 减半看效果。其次考虑增大 LoRA 的 rank（如果用 LoRA）或者加 dropout，增加模型的泛化能力。如果还不行，审视训练数据——可能数据多样性不足。

Q13. vLLM 部署 32B 模型时，面对突发流量导致的 KV Cache 溢出，除了 PageAttention，在应用层如何做请求优先级降级？

A13.

PagedAttention 解决了内存碎片问题，但总量不够时仍会溢出。应用层的降级策略：

1. 请求优先级队列 + 抢占。定义优先级：交互式请求 > 实时分析 > 批量任务。KV Cache 接近水位线时，低优先级请求进入等待队列，不分配 KV Cache。已经在推理中的低优先级请求可以被抢占（vLLM 本身支持 preemption），释放 KV Cache 给高优先级请求。

2. 动态 max_tokens 缩减。监控 KV Cache 使用率。达到 80% 时，新请求的 max_tokens 自动缩减（比如从 4096 降到 2048）。用户可能拿到略短的回复，但不会被拒绝服务。

3. 请求级的 context 压缩。入队前用一个轻量模型（如 Haiku 级别）对长 prompt 做压缩——提取关键信息，缩短 input tokens。这直接减少 KV Cache 占用。

4. 多实例负载均衡 + 溢出路由。单实例扛不住时，将溢出请求路由到备用实例。如果所有实例都满载，降级到更小的模型（32B → 7B）处理低优先级请求，保证高优先级请求仍然用 32B。

5. 批量请求的延迟调度。对非实时的批量任务，不立即执行，而是等到 KV Cache 有余量时才调度。用一个简单的令牌桶限流器控制批量任务的并发数。

# 产品架构与说明

## 1. 问题定义

当前痛点不是“agent 数量不够”，而是“协作过程没有被程序化”。

传统做法的问题：

- 长会话单 agent 容易把旧上下文、妥协意见和无关历史一起带入后续实现，导致项目漂移。
- 使用共享文件夹和文档邮箱虽可审计，但无法强制流程，也无法自动判断现在该谁继续做。
- 用户需要手动打开多个 CLI，轮流触发双方读取文档、回应意见，消息总线被用户自己承担，效率很低。

本项目要解决的是：

- 把 agent 协作从“自由对话”改成“阶段化工作流”
- 把共享文档从“流程真相”降级为“产物本体”
- 把用户从“手动转发者”提升为“可视化监督者和审批者”

## 2. 产品目标

### 2.1 目标

- 在本地统一编排多个 CLI agent 的工作流
- 强制执行 `计划 -> 评审 -> 修订 -> 审批 -> 实施 -> 复核 -> 验证 -> 结项`
- 自动沉淀结构化产物、评审意见和逐条回应
- 让用户在页面中实时查看进度、争议点、风险点和审批关卡
- 在高风险或意见僵持时自动触发第三角色参与

### 2.2 非目标

- 不追求一次性变成远程团队协作平台
- 不把所有代理都做成永久在线人格
- 不让“优化人”默认常驻，以免增加延迟和上下文膨胀
- 不先做复杂权限系统、多租户和云部署

## 3. 用户与角色

### 3.1 用户

- 个人开发者或小团队负责人
- 同时使用多个 coding agent，希望项目不跑偏
- 需要在本地敏感代码仓中运行，不愿把全部上下文迁移到外部平台

### 3.2 角色模型

#### Executor

- 负责计划产出、代码修改、问题修复
- 必须逐条回应 Reviewer 的 finding
- 在未过关前不得直接越过评审关口

#### Reviewer

- 负责对计划和实现进行严格审核
- finding 分为 `blocker / concern / suggestion`
- 重点指出风险、遗漏、回归、约束不满足，而不是润色措辞

#### Verifier / Judge

- 不是默认常驻角色，而是条件触发
- 负责自动验证、仲裁分歧、检查计划与实现是否一致
- 优先承担“裁判”而不是“优化人”

#### Human Gate

- 用户本人
- 不参与每一步中转，但可以在关键关卡审批、驳回、加批注
- 需要能随时查看进度、差异、阻塞项和争议历史

## 4. 为什么不是固定三人组

一个执行人和一个监督人对大多数任务足够，但要补一个条件触发的第三角色。

常驻三人组的问题：

- 每轮都多一跳，整体节奏明显变慢
- “优化人”容易和执行人职责重叠
- 对低风险任务会形成过度治理

建议的规则是：

- 默认：`Executor + Reviewer`
- 条件触发 `Verifier / Judge` 的情况：
  - 计划修订超过 2 轮还未收敛
  - 涉及数据库、架构、并发、安全等高风险主题
  - Reviewer 与 Executor 对同一条意见持续冲突
  - 自动验证失败且双方解释不一致

## 5. 产品形态

### 5.1 核心形态

一个本地优先的“代理工作流编排器”。

它包含：

- 后端编排服务：管理项目、运行实例、阶段流转、产物索引、审批状态
- CLI adapter：接入 `Codex CLI`、`Claude Code CLI` 等本地 agent
- 产物库：保存 Markdown、JSON、日志、补丁摘要、测试报告
- 前端控制台：以状态机视图展示项目进度和争议

### 5.2 用户看到的不是文档目录

页面应展示：

- 当前阶段：现在在计划、评审、修订、实施还是等待你审批
- 时间线：每一步是谁发起、谁回复、结论是什么
- 产物列表：计划草案、review 报告、逐条回应、验证报告
- 差异视图：这一轮相对上一轮具体改了什么
- 争议面板：哪些 blocker 已解决，哪些还未关闭
- 审批面板：哪些节点必须由用户批准

## 6. MVP 功能范围

### 6.1 必做

- 本地创建项目和 workflow run
- 角色配置与 workflow 模板加载
- 计划评审闭环
- 实施评审闭环
- 验证和最终人工关卡
- 结构化 finding 跟踪
- 产物归档与索引
- 用户可视化面板

### 6.2 延后

- 多用户协作
- 远程 agent 集群
- 复杂权限系统
- 向量搜索和自然语言检索历史产物
- 自动拆任务和并行分配多个执行 worker

## 7. 存储原则

不要在“只用文档”与“只用数据库”之间二选一，应采用混合模式。

### SQLite 负责

- 当前阶段与状态
- 项目、运行实例、阶段、审批、finding 的关系
- 哪些意见已关闭，哪些仍阻塞
- 前端查询、筛选、统计、提醒所需的结构化数据

### 文件产物库负责

- Markdown 计划书
- Review 报告
- Executor 逐条回应
- JSON 检查结果
- 测试日志与补丁摘要

一句话概括：

- 文档负责可读
- 数据库负责可控

## 8. 关键成功指标

- 用户不再需要同时盯两个 CLI 窗口做人工转发
- 任意时刻都能知道项目卡在哪个阶段、阻塞项有哪些
- 同一轮评审中的每条 blocker 都有可追踪回应
- 高风险任务能自动引入第三角色而不是靠用户临时拍脑袋
- 相比文档邮箱模式，单轮协作耗时明显下降

## 9. 风险与约束

- 如果 agent 输出格式不稳定，结构化记录会变脆弱，因此需要强制模板
- 如果让所有角色都长时间保持会话，仍会出现上下文膨胀，因此建议阶段性 worker
- 如果 UI 只是文档目录镜像，用户依旧会失去全局感
- 如果把“优化人”做成默认角色，会明显拖慢低风险任务

## 10. 第一阶段交付建议

第一阶段只验证以下链路：

1. 创建 run
2. Executor 提交计划
3. Reviewer 产出 finding
4. Executor 逐条回应并修订
5. 用户批准后开始实施
6. Reviewer 复核实现
7. Verifier 记录验证结果
8. 用户做最终结项审批

只要这个闭环稳定，再接真实 CLI adapter 和实时前端推送。
